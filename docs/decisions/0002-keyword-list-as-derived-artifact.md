# ADR-0002: Protected-keyword list is a derived artifact (not hand-maintained)

**Status:** Accepted (2026-06-21)
**Deciders:** Mavin + Claude
**Triggered by:** 3 rounds of evidence that hand-typed lists/counts are unreliable

---

## Context

The protected-keyword list (which identifiers force RED classification if touched) was originally proposed as a hand-written array in Mavis's v3 response. Over 3 rounds:
- v3: claimed 47 entries → actually 86 (with 53 invented)
- v4: claimed 78 entries → actually 99 (hand-typed undercount)
- v5: claimed "clean 99" → actually malformed 94 (script bug)

The pattern: a model hand-typing a list/count is unreliable. The blind spot moves, but it's always there.

## Decision

The protected-keyword list is **generated by a script** (`scripts/keyword-extractor.js`). The script:
1. Reads the baseline HTML
2. Greps for real functions + identifiers
3. Emits the array AND the count together (exit code = count)
4. Writes the result to `scripts/PROTECTED_KEYWORDS.js`

Hand-edits to the keyword list are **forbidden**. The script is the source of truth.

## Rationale

**Why derived artifact:**
1. **Cannot invent names** — the script greps the real file, only real identifiers get emitted
2. **Cannot miscount** — the count is a script output, not a hand-typed number
3. **Auto-updates per baseline** — when the baseline changes, regenerate the keyword list
4. **Auditable** — the script is in the repo, you can read what it does

**Why not hand-maintained:**
- 3 rounds of evidence it fails
- "Show work, not verdicts" — a hand-typed count is a verdict without the work
- Self-test on a known case before scaling — applies to lists, not just code

## Trade-offs

**Gained:**
- Always-accurate count (script output, not hand-typed)
- Cannot invent or miss names (grep guarantees real)
- Audit trail (the script is reviewable)
- Auto-updates when baseline changes

**Lost:**
- Slightly more setup (need Node.js to regenerate)
- Script can have its own bugs (but the test suite catches them)
- Hand-curation of "what should be a keyword" is harder (currently fully automated)

## The implementation

```bash
# Regenerate from any baseline
node scripts/keyword-extractor.js <baseline.html> scripts/PROTECTED_KEYWORDS.js

# Run the test suite to verify the script works
npm test
```

The script has 11 vitest tests that verify:
- Self-test on minimal fixture
- Numeric count emitted
- Exit code matches count
- No junk like `'function name('`
- Anchored matching for bare-word utilities
- Real-baseline integration tests
- Gate integration tests
- Consolidation review tests

## When to revisit

This decision should be revisited if:
- The methodology needs hand-curated exceptions (e.g., "this identifier is special-cased")
- The script becomes too complex to maintain
- A better tool for managing derived artifacts emerges (e.g., build systems)

## Evidence

The 3 rounds that triggered this ADR:
- v3: hand-typed 47, actual 86, 53 invented → caught by Claude's grep
- v4: hand-typed 78, actual 99 → caught by Claude's count
- v5: claimed script output clean 99, actual malformed 94 → caught by Claude running the script

The pattern is consistent: hand-typed is wrong, script output is right. The ADR codifies the lesson.
